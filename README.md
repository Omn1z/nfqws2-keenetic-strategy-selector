# NFQWS2 Strategy Selector для Keenetic

Веб‑инструмент для **автоматического подбора рабочих стратегий обхода DPI** для
[`nfqws2-keenetic`](https://github.com/Anonym-tsk/nfqws-keenetic) (движок
[zapret2](https://github.com/bol-van/zapret2), Lua). Запускается прямо на роутере,
тестирует стратегии в несколько потоков на ваших списках доменов/IP, сохраняет
только те, что реально работают (есть ответ и нет обрыва на 16 КБ), и сортирует
их по скорости.

> ⚠️ Требуется уже установленный пакет **`nfqws2-keenetic`** (бинарь
> `/opt/usr/bin/nfqws2`, движок zapret2 v0.9.x) и Entware с `iptables`.

## Возможности

- **Веб‑интерфейс** — один статический бинарник со встроенным UI, без зависимостей.
- **Дашборд и мониторинг** — главная страница со сводкой (статистика TG WS Proxy,
  активные соединения, пакеты DPI‑движка очереди NFQUEUE, трафик WAN) и **живыми
  графиками нагрузки**; вкладка «Соединения» (живой conntrack с фильтром и
  сортировкой) и «Активность устройств» — что устройства LAN не могут открыть
  (кандидаты на подбор стратегий, отправляются в прогон одной кнопкой). Для отладки
  соединений у каждого устройства — **трассировка 30 с** (таймлайн conntrack) и
  **захват пакетов `.pcap`** (`tcpdump`, ставится по запросу), открываемый в Wireshark.
- **Логи** — живой просмотр логов сервиса по модулям (включая TG WS Proxy), с
  фильтром, паузой и авто‑прокруткой.
- **Система** — отключение авторизации (с подтверждением) и логирования.
- **Списки** доменов и IP/CIDR; к каждому списку прикрепляются найденные ранее
  рабочие («базовые») стратегии.
- **Прогоны в несколько потоков** — выбираете число потоков, стратегии тестируются
  параллельно (каждый поток — изолированная песочница).
- **Изоляция трафика** — основной сервис nfqws2 продолжает работать, тестовый
  трафик отводится отдельными правилами iptables (сеть не нарушается во время прогона).
- **Метрики и сортировка** — задержка (TTFB), пропускная способность (КБ/с) и
  комбинированный «коэффициент скорости»; таблица сортируется по любой колонке.
- **Свои блобы** — загрузка собственного fake‑payload, **создание `ClientHello`**
  (снять с трафика устройства через `tcpdump` или сгенерировать по домену) и
  **корзина** (удаление в корзину с восстановлением).
- **Применение** найденной стратегии в основной конфиг nfqws2 (с резервной копией).
- **Авторизация** по логину/паролю роутера (как в nfqws-keenetic-web). Отключение — `N2S_NOAUTH=1`.
- **Выбор блобов для прогона** — отмеченные блобы загружаются во все тестируемые стратегии.
- **GeoSite/GeoIP** — загрузка `geosite.dat`/`geoip.dat` или текстового списка и импорт категории в тестовый список.
- **DNS (DoH/DoT)** — список шифрованных DNS (Google, Cloudflare, Comss, Yandex,
  xbox-dns, GeoHide, AdGuard и свои). В прогоне можно прогнать стратегии через
  каждый выбранный DNS — в результатах видно, какой DNS открыл доступ.
- **TG WS Proxy** — встроенный Telegram MTProto → WebSocket прокси прямо на
  роутере (порт DPI-устойчивого подключения к Telegram через WSS), отдельной
  вкладкой; по умолчанию выключен.
- **Управление движком NFQWS2** (вкладка «Сервисы → NFQWS2») — редактирование
  конфига (`nfqws2.conf`: быстрые настройки + полный текстовый редактор),
  Lua‑скриптов обхода и списков доменов/IP прямо в панели (загрузка/скачивание/
  удаление, дедуп списков, защита критичных файлов). Показ версии движка рядом с
  версией пакета и обновление движка по `opkg` при наличии нового релиза.
  «Применить (reload)» по `SIGHUP` перечитывает списки без обрыва очереди.
  Полноценная замена стороннего `nfqws-keenetic-web`.
- **Установщик / обновление / удаление**, мульти‑арч сборки в GitHub Releases.

## Установка

```sh
curl -fsSL https://raw.githubusercontent.com/Omn1z/nfqws2-keenetic-strategy-selector/master/packaging/install.sh | sh
```

После установки откройте `http://<IP‑роутера>:8090`.

Сервис ставится как `/opt/etc/init.d/S52nfqws2-strategy` (автостарт), данные — в
`/opt/etc/nfqws2-strategy/`.

## Использование

1. **Списки** → создайте список, вставьте домены (по одному в строке), сохраните.
2. **Прогон** → задайте число потоков, при желании выберите конкретные стратегии
   (по умолчанию — все), нажмите «Запустить». Прогресс и результаты обновляются вживую.
3. **Результаты** → таблица сортируется по задержке / скорости / коэффициенту.
   Рабочие стратегии (зелёные, прошли все цели) автоматически сохраняются в список.
4. **Применить** → понравившуюся стратегию можно записать в `nfqws2.conf`
   (создаётся бэкап `*.n2s.bak`) и при желании перезапустить сервис.
5. **Блобы** → вкладка для загрузки своих fake‑payload; путь подставляется в
   стратегию как `--blob=имя:@/путь`.

## Как это работает

Каждый поток («песочница») получает отдельный номер очереди NFQUEUE, диапазон
исходных портов и временные цепочки iptables. Тестовые соединения помечаются
connmark `0x20000000`, поэтому основной nfqws2 (очередь 300) их игнорирует, а
тестовый экземпляр nfqws2 с проверяемой стратегией обрабатывает только их.
HTTP‑клиент скачивает тестовый URL через песочницу, измеряет байты/время и ловит
обрыв на ~16 КБ. По завершении все правила и процессы убираются.

## Сборка из исходников

Нужны Go ≥ 1.22 и Node ≥ 20 (веб-интерфейс — React 19 + TypeScript + Tailwind на
Vite). Скрипты сборки сами собирают фронтенд (`npm`) в один встраиваемый файл,
а затем линкуют бинарь.

```sh
sh scripts/build.sh v1.0.0        # Linux/macOS
powershell -File scripts/build.ps1 -Version v1.0.0   # Windows
```

Бинарники появятся в `dist/` для `arm64`, `arm` (v7), `mipsle`, `mips`.

## Поддерживаемые архитектуры

`aarch64` (arm64), `armv7` (arm), `mipsel` (mipsle), `mips`. Установщик определяет
архитектуру через `opkg print-architecture`.
